# AI协作质量管理框架

本skill提供一套通用的人机协作质量管理体系，适用于任何需要AI深度参与的项目（需求撰写、文档体系搭建、分析报告、方案设计等）。核心目标：**让AI输出的可信度透明、让协作质量可持续提升**。

---

## 何时使用

- 启动任何新项目时 → 初始化协作基础设施
- 多轮迭代协作中 → 执行AI自检、版本管理
- 协作session结束时 → 执行复盘机制
- 发现AI输出有问题时 → 更新错误自检表

---

## 框架概览

本框架包含四大机制，形成闭环：

```
协作 → AI自检（每次输出前） → 复盘（每次session后） → 迭代（更新规则/边界） → 下次协作质量↑
```

详细说明见 `references/collaboration-mechanisms.md`。

---

## 机制一：协作边界声明

在项目启动时，建立AI能力边界声明。

### 执行步骤

1. 在项目核心文档（或独立的协作规范文档）中创建「AI协作边界声明」章节
2. 根据项目领域，填写以下模板：

```markdown
### ⚠️ AI协作边界声明

| AI做不到的（需要用户提供） | 原因 | 建议来源 |
|--------------------------|------|---------|
| {根据项目领域识别}        |      |         |

| AI能做的 | 说明 |
|---------|------|
| {根据项目领域识别} |  |

**协作规则**：
1. AI输出中使用的每个事实性断言，必须标注来源：`[AI训练数据,待验证]` / `[用户提供,{日期}]` / `[需要用户提供]`
2. 需要外部数据的任务，AI必须先列出「数据需求清单」，等用户填入后再生成完整输出
3. 如果用户未提供必要数据，AI不得自行填充，必须留空并标注 `[需要用户提供]`
```

3. 边界声明是**动态的**——每次发现新的能力边界时立即更新

## 机制二：AI反思自检表

每次生成输出前，逐条对照自检。

### 通用错误模式（适用于所有项目）

| # | 错误类型 | 自检问题 | 违反后果 |
|---|---------|---------|---------|
| E1 | **虚假精确** | "我输出的具体数字/数据，来源是什么？是训练数据还是用户提供的？" | 用户基于不可靠数据做决策 |
| E2 | **沉默填充** | "有没有我查不到/不确定的信息？我是否已全部标注？" | 用户无法区分可靠信息与AI猜测 |
| E3 | **伪权威推荐** | "我推荐这个工具/方案/方法的依据是什么？是验证过的还是听说的？" | 用户信任未经验证的推荐 |
| E4 | **未验证的逻辑断言** | "这个判断有实证支持吗？还是仅仅逻辑上说得通？" | 看似合理但实际无效的方案 |
| E5 | **过度自信** | "我的确定程度真的有这么高吗？有该说'我不确定'的地方吗？" | 用户误判AI输出的可靠程度 |

### 自检执行规则

1. **生成任何正式输出前**：逐条过E1-E5，命中 → 在输出中明确标注
2. **推荐工具/方案/方法时**：标注依据强度 `[已验证]` / `[逻辑推断]` / `[社区口碑]` / `[不确定]`
3. **给出知识性判断时**：确定程度低于80% → 主动说明"这一点我不够确定，建议交叉验证"
4. **发现新错误模式时**：主动提出追加到自检表中

### 项目特定错误

随着协作深入，将项目中实际发生的错误追加到自检表。格式：

```
| E{n} | **{错误名称}** | "{自检问题}" | {违反后果} |
```

## 机制三：版本迭代管理

### 迭代优化池

在项目中维护一个「迭代优化池」文档，记录体系的不足与改进历史。

#### 结构模板

```markdown
# {项目名} · 迭代优化池

记录当前体系的不足与已完成的迭代，按优先级排序。

## 一、已知不足/待补充方向

### 1. {问题描述} {优先级标签}
- **问题**：...
- **影响**：...
- **待补充**：...

## 二、迭代总览

| # | 迭代项 | 优先级 | 状态 |
|---|--------|--------|------|
| 1 | ... | ⭐高 / 中 / 低 | 待开始 / 进行中 / ✅ 已完成（简述） |
```

#### 状态管理规则

- 每次变更后立即更新状态
- 状态标注包含简要说明（不只是"完成"，要说明做了什么）
- 新发现的问题随时追加

### 版本号管理

- 核心文档使用版本号（如v1.0, v1.1...）
- 每次功能性变更 → 版本号递增 + 迭代优化池记录
- 版本号写在文档头部，方便追溯

## 机制四：协作复盘

### 触发时机

每次协作session结束前，AI应主动提醒："是否需要做本次复盘？"

### 复盘维度

| 维度 | 内容 | 执行方 |
|------|------|--------|
| **本次做了什么** | 列出所有变更（文档/代码/方案） | AI整理 |
| **哪里做得好** | 有效的环节、好的追问 | 双方讨论 |
| **哪里出了问题** | 错误、遗漏、不准确 | 双方讨论 |
| **新错误模式** | 是否需要新增E类自检项？ | AI自查 + 用户反馈 |
| **边界调整** | AI能力边界表是否需要更新？ | 双方讨论 |
| **迭代记录** | 写入迭代优化池 | AI执行 |

### 产出物

1. 迭代优化池更新
2. 复盘报告（如有重大变更，存入output/）
3. 自检表更新（如发现新错误模式）
4. 边界声明更新（如发现新能力边界）

## 跨文档关联管理

当项目包含多个互相引用的文档时：

### 关联同步规则

1. 任何文档变更后，检查是否有其他文档引用了变更内容
2. 如有 → 同步更新所有关联文档
3. 在迭代优化池中记录"本次同步了哪些文档"

### 建议的关联记录方式

在项目导航文档或迭代优化池中维护一张关联表：

```markdown
## 文档关联表
| 源文档 | 关联文档 | 关联内容 |
|-------|---------|---------|
| A.md  | B.md    | A中的X规则在B中被引用 |
```

## 待办清单管理

对于需要用户执行的行动项，维护一个结构化待办清单：

### 结构要求

1. 按优先级分组
2. 每项包含：待办内容、去哪做、验证什么、状态（□ / ✅）
3. 明确标注"完成后请告诉AI结果，以便同步更新"

### 协作流程提醒

在待办清单中包含关键协作流程说明，避免用户遗忘。例如：
- 提醒用户在让AI做某类任务前需要先提供什么
- 提醒用户如何判断AI输出是否可靠

---

## 新项目启动Checklist

启动新项目时，按以下顺序初始化协作基础设施：

```
□ 1. 明确项目范围和目标
□ 2. 创建AI协作边界声明（识别AI能做/不能做的事）
□ 3. 初始化迭代优化池
□ 4. 根据项目领域定制E类自检表（在通用E1-E5基础上追加）
□ 5. 创建项目导航文档（如项目包含多个文档）
□ 6. 创建待办清单（如有需要用户执行的项）
□ 7. 开始协作，每次输出前执行自检
```
