# 数据源提供逻辑（API / MCP / Skill）

> **版本：v1.0**
>
> **定位**：本项目所有“依赖数据”的地方，都必须能回答一句话：**这个数字从哪里来，能不能复现查询？**
>
> 目标是让LLM只做“分析与写作”，不做“编数据”。数据要么来自你手工提供（带来源与日期），要么来自可查询的工具（API / MCP / skill）。

---

## 一、核心原则（硬规则）

1. **不允许LLM自行填充任何会影响买卖决策的数字**（价格、估值、财务、分位、宏观指标、预期等）。  
   允许LLM输出“需要你提供”的数据清单与计算公式，但**不得代填结果**。
2. **每个关键数字必须可复现查询**：至少包含 `来源` + `日期` + `查询方式`（手工/API/MCP/skill）+ `字段口径`（TTM/季度/年化/等权/中位数等）。
3. **预测类数据也不能靠LLM凭空推理**：  
   - 优先使用：公司指引、卖方一致预期、权威机构预测、可检索新闻/公告  
   - 必须标注：`[来源,{日期}]`，并写清楚“预测是谁的预测”  
4. **交叉验证**：任何影响“买/卖/仓位”的关键数字，至少用2个独立来源交叉确认；冲突未解决前，不进入结论。

---

## 二、数据契约（Data Contract：你要什么数据）

> 最小化原则：只采集“会改变决策”的数据。采集之前先问：**没有它我会做出不同决定吗？**

| 数据类别 | 决策用途 | 关键字段（最小集合） | 允许来源（示例） | 推荐工具形态 |
|---------|----------|----------------------|------------------|--------------|
| 行情/交易 | 估值、仓位、执行 | 当前价格、币种、时间戳、市值（可选） | 券商/行情终端；交易所公开数据 | **API/MCP优先**；无则手工提供 |
| 估值指标 | 安全边际、分位判断 | PE/PB/PS/EV-EBITDA、历史分位区间、口径说明 | 理杏仁/富途/中证/交易所等 | **API/MCP/skill优先**；无则手工提供 |
| 财报数据 | 质量验证、生存判断 | 营收、净利、经营现金流、自由现金流、现金、负债、关键经营指标 | 年报/季报/10-K/20-F/公告 | **官方披露优先**；API/MCP抓取或手工摘录 |
| 宏观/利率/波动 | 股权上限、对冲触发 | CPI/PMI/收益率曲线、VIX等 | 统计局/央行/权威数据库 | **API/MCP优先** |
| 预测/预期 | 情景假设、压力测试 | 公司指引、卖方一致预期、机构预测、新闻事件 | 公告/机构研报/新闻 | **可检索来源**（API/MCP/手工引用） |

> **注意**：此表不等于“推荐某个平台”。任何第三方平台数据都可能存在口径差异，必须写清口径并交叉验证。

## 三、工具形态（怎么拿到数据）

### 3.1 手工（最低可行）

你从富途/理杏仁/交易所/公告等查到数据，按统一格式贴给AI，并带上：

- 数据项名称 + 口径（例如：PE-TTM / PE中位数）
- 数值 + 币种/单位
- 来源 + 查询日期（最好附页面路径/截图说明）

### 3.2 API（可复现、可自动化）

当某个数据项需要频繁更新或用于硬规则触发（如分位、VIX、收益率）时，优先API化。  
本项目不绑定具体供应商，但要求：**字段口径明确、可复现查询、允许记录原始响应**。

### 3.3 MCP（工具调用：让LLM“用工具查”，而不是“凭记忆猜”）

当你有可用的MCP数据服务时，LLM只能通过MCP读取数据（并记录调用参数与返回结果）。  
建议按数据域拆分MCP：`market-data` / `macro-data` / `filings-data` / `consensus-data`。

### 3.4 skill（流程固化：让每次协作都走同一条路）

skill用来固化三类输出：

1. **数据需求清单生成**（含“获取方式：手工/API/MCP/skill”）
2. **数据一致性审计**（同一字段多源冲突时的对照表）
3. **报告生成**（只消费已提供/已查询的数据，不得补数）

---

## 四、标准工作流（强制执行）

1. **LLM先输出“数据需求清单”**（不得出现具体数字）  
   - 每条数据必须写：字段名、用途、口径、获取方式（手工/API/MCP/skill）
2. **数据提供者补齐数据**（你手工提供或工具自动拉取）  
   - 每条数据带：来源、日期、口径说明
3. **LLM生成报告**  
   - 所有数字必须标注 `[用户提供,{日期}]` 或 `[工具获取,{日期}]` 或 `[需要你提供]`
4. **LLM做一次“数据审计”**  
   - 列出关键数字清单、来源、是否已交叉验证、是否存在冲突/缺失

---

## 五、边界：LLM允许做什么、不允许做什么

**允许**：
- 用你提供/工具获取的数据做逻辑推导、情景分析、压力测试（假设必须写清楚）
- 做定性研究框架、写作、结构化输出

**不允许**：
- 生成任何关键数据的“看起来合理”的数值（包括“历史均值”“大概区间”等）并用于决策
- 把自己的推理当作“预测数据”输出（预测必须来自可检索来源）
